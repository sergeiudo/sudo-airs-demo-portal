Part 1 — Mental model
Mental model

The one design decision that makes it work

One key + per-request config

The portal authenticates with one key (sudo-demo-portal-full-key) and tells Portkey which config to use on each request. That requires Allow Config Override = ON on the key.

With override OFF, the request's config is ignored and the key is locked to one config — the 3-lane switch would collapse to a single lane.

Configs are the policy unit

  • Integration = credentials only
  • Guardrail = reusable check (AIRS, PII, words)
  • Config = provider + which guardrails + cache + retry
  • App swaps config = swaps policy. No code change.

Per Portkey docs: a request config overrides the key's default unless "Allow Config Override" is turned off.

Part 4 · Guardrail A

sudo-portkey-guardrails — native policy

id pg-sudo-p-c7e4ad · attached to the AI-GW native config on input + output

No PII REDACT

Detects & masks to **** before the model sees it:

EMAIL · PHONE · ADDRESS · NAME · IP · CREDIT_CARD · SSN

Contains Code

Format: Python

Inverse = ON → check fails (blocks) when the content contains Python code.

Contains (banned)

Words: סרגיי · Chekpoint · Kokomoko

Operator = none → fails (blocks) when any of these appears.

→ These are business / data policy controls — org-specific, not threat detection. AIRS doesn't know your banned-word list.

Part 6

What each flow does

No protection → AI Gateway → AI Gateway + AIRS.

Flow 1 · No gateway

Direct to the model — no inspection

What happens

  • The portal calls Vertex directly with its own GCP creds — Portkey is not in the path.
  • No config, no key, no guardrail.
  • This is the "before" picture: whatever the model does is all you get.

In the code

portkey-routes.js → callDirectVertex(). For gemini-3.x it uses the Vertex OpenAI-compatible endpoint on global with a google/ prefix; gemini-2.5 uses the Gemini SDK on us-central1.

Demo result: PII is echoed back raw · injections reach the model · only the model's own refusal (if any) stops anything.

Part 6 — Flows
Flow 2 · AI Gateway (native)

Portkey gateway + native guardrails

What happens

  • Request goes through Portkey with config pc-sudo-d-a72af9.
  • Input scanned, then model, then output scanned — both stages run portkey.pii, containsCode, contains.

What you'll see

  • PII → REDACTED (masked to ****), request proceeds
  • Banned word / Python code → BLOCKED
  • Prompt injection / jailbreak → allowed (native can't see it)

"Allowed" is orange in the demo (= got through). Redaction shows as a blue ALLOWED · REDACTED badge.

Part 6 — Flows
Flow 3 · Gateway + AIRS

Portkey gateway + Prisma AIRS

What happens

  • Request goes through Portkey with config pc-sudo-t-5a505b.
  • Portkey hands input + output to AIRS (profile sudo-airs-api-profile-new) for scanning.
  • If AIRS returns action=block, the request is stopped before the model.

What you'll see

  • Prompt injection · jailbreak · dangerous · URL · toxic → BLOCKED
  • PII generation → BLOCKED (AIRS is stricter than redact)
  • Benign · banned word · benign code → allowed (no security threat)

Part 9 · Key learnings

Gotchas worth remembering

  • Allow Config Override must be ON on the portal key — otherwise per-request config switching is ignored and all lanes collapse to one.
  • gemini-3.x is global-only here; the direct lane routes it via the Vertex OpenAI endpoint with a google/ prefix. gemini-2.5 is us-central1.
  • Remove override_params from configs so the request's model wins (model picker drives all lanes).
  • Guardrail semantics: Contains Code Inverse=ON = block on code present · Contains operator=none = block on word present · No PII = redact (allows, masks).
  • Provision the model in Portkey (Integration → Model Provisioning) and enable it in Vertex — both gates must pass, else model_not_allowed / 404.
  • Redaction shows as ALLOWED · REDACTED (Portkey sets transformed:true on the hook).
