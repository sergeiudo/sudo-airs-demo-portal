STEP 3 / 10

Guardrail — Prisma AIRS intercept

Guardrails→+ Create→search"Prisma AIRS"
  1. Plugin credentials (first time only): paste your Prisma AIRS API key — the AIRS_API_KEY value from .env.
  2. Add check “PANW Prisma AIRS Intercept” with the parameters below.
  3. Under Actions: toggle Deny ON · async OFF. Deny is what converts action=block into a hard request rejection (verdict:false + the hooks_failed exception the backend normalises).
  4. Name it sudo-airs · Save · note the pg-… ID — needed in Steps 5 & 7.

STEP 5 / 10

No provider pin, no override_params (old config had both). The app always sends full @provider/model, so dropping the pin is what lets Bedrock models route through this guardrailed lane.

STEP 8 / 10

Service API key

FieldValue
TypeService — NOT User

Scopescompletions/chat (required) + logs.view, logs.list, analytics.view, configs.read (nice for API-side debugging)

Today's lesson, applied: a service key belongs to the workspace, not a person — it survives membership changes (that's exactly why the old demo kept working after your removal). In your workspace that's now a feature: the demo never dies because of personnel churn. A user-type key would die with its user.
